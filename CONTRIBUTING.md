# Contributing to the H.I.R.E. Framework


Thank you for your interest in contributing! This framework is **community-driven**, and we need your help to make it better.


## How You Can Contribute


### 1. Share Your Case Studies
Have you built an AI agent? We want to hear about it!


**What we're looking for**:
- Real-world implementations (successes AND failures)
- Metrics you tracked and results you achieved
- Lessons learned and what you'd do differently
- Industry-specific adaptations of the framework


**How to contribute**:
1. Copy the [case study template](/case-studies/template.md)
2. Fill in your story
3. Submit a pull request to `case-studies/`


### 2. Improve Metric Definitions
Help us refine how we measure AI agent success.


**What we need**:
- Better formulas for existing metrics
- New metrics we haven't thought of
- Industry-specific metric variations
- Code templates for analytics tools


**How to contribute**:
1. Open an issue describing the metric improvement
2. Submit a pull request with updated documentation
3. Include code examples if applicable


### 3. Improve Code Examples
Make technical implementations better.


**What we need**:
- More robust error handling
- Performance optimizations
- Additional programming languages
- Unit tests and documentation


**How to contribute**:
1. Fork the repo
2. Improve code in `framework/agent-metrics` (coming soon)
3. Submit a pull request with clear explanations


### 4. Challenge Our Assumptions
Tell us where we're wrong!


**What we want to hear**:
- *"This metric doesn't work in [industry/context]"*
- *"You're missing [important consideration]"*
- *"This approach has a fatal flaw: [explanation]"*
- *"Here's a better way to think about [concept]"*


**How to contribute**:
1. Open a GitHub issue with your point of view
2. Provide specific examples or evidence
3. Suggest improvements


## Contribution Guidelines


### Before You Start


1. **Check existing issues** to avoid duplicate work
2. **Open an issue first** for major changes (so we can discuss approach)
3. **Keep it practical**: focus on real-world applicability


### Code Contributions


**Style**:
- Python: Follow PEP 8
- Markdown: Use clear headings and formatting
- Comments: Explain *why*, not just *what*


### Documentation Contributions


**Style**:
- Use clear, concise language
- Include examples and visuals where helpful
- Assume the reader is smart but unfamiliar with the topic


**Structure**:
- Start with "Why this matters"
- Provide step-by-step instructions
- End with "Next steps" or "Related resources"


### Case Study Contributions


**Required sections**:
1. **Problem**: What pain point were you solving?
2. **Solution**: What did you build?
3. **H.I.R.E. Evaluation**: How did you evaluate the agent?
4. **Results**: What metrics did you track? What were the outcomes?
5. **Lessons Learned**: What worked? What didn't?


**Optional sections**:
- ROI analysis
- Technical architecture
- Team structure
- Timeline

## Pull Request Process


### 1. Fork and Clone
```bash
git clone https://github.com/[your-username]/hire-ai-framework.git
cd hire-ai-framework
git checkout -b feature/your-feature-name
```


### 2. Make Your Changes
- Follow the guidelines above
- Test your code
- Update documentation


### 3. Commit with Clear Messages
```bash
git add .
git commit -m "Add: [clear description of what you added]"
```


**Commit message format**:
- `Add: [new feature/content]`
- `Fix: [bug or error]`
- `Improve: [enhancement to existing content]`
- `Docs: [documentation changes]`


### 4. Push and Create PR
```bash
git push origin feature/your-feature-name
```


Then create a pull request on GitHub with:
- **Clear title**: "Add case study: financial fraud AI agent"
- **Description**: What you changed and why
- **Related issues**: Link to any relevant issues


## Recognition


Contributors will be recognized in:
- README.md (Contributors section)
- Release notes
- Conference presentations (if applicable)
